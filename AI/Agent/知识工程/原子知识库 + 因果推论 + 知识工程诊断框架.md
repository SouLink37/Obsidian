# 原子知识库 + 因果推论 + 知识工程诊断框架

> **一句话定义**：一个反 LLM / 人类共同诊断弱点的工程化知识系统，把诊断推理拆到**最小可验证单位**（原子知识），用**因果图**组合而非决策树枚举，强制每条结论标注"不证明什么"、"可被反驳的条件"和"适用范围"，并把案例经验沉淀流程结构化（候选 → 假设 → 已接受 → 废弃）。

---

## 一、问题与动机

### 1.1 LLM 和人类共同的诊断弱点

| 弱点 | 表现 |
|---|---|
| **过早收敛**（premature closure）| 看到第一个强证据就锁定根因，不再考虑替代假设 |
| **共现谬误**（co-occurrence fallacy）| 把"X 和 Y 同时出现"写成"X 导致 Y" |
| **症状当根因**（symptom-as-cause）| 把用户报告的现象直接归到底层硬件/软件 |
| **修复偏倚**（confirmation bias by fix）| "重启好了 → 就是 X 问题"，但没验证因果链 |
| **过度泛化**（overgeneralization）| 一个 case 成功就写成通用规律 |
| **共时混淆**（temporal confusion）| 不区分 UTC / 本地时间 / 不同采样窗 |
| **跨域跳跃**（cross-layer leap）| App 症状直接归因到底层硬件，跳过中间链路 |

LLM 还有特有弱点：
- **锚定效应**（anchoring）：看到 FSE/Dev 的结论后无法独立判断
- **流畅性偏倚**（fluency bias）：能写出流畅推理就误以为推理正确
- **训练数据残留**（spurious correlations from training）：错把训练里的共现当因果

### 1.2 传统 SOP 决策树的局限

```
SOP 决策树：
  if 症状 X:
    if 字段 A < 阈值: → 换 B 件
    else: → 换 C 件
  elif 症状 Y: ...
```

**问题**：
- **覆盖不全**：写不完所有故障家族（用户的"是否所有问题都覆盖"永远是 No）
- **过拟合一次成功**：单次有效就固化为规则
- **不可演化**：版本/硬件变了，整棵树要重写
- **缺反例机制**：决策树不天然带"X 不证明 Y"
- **跨域不复用**：每个领域要写自己的树
- **审计困难**：决策路径不显式列出"为什么排除其他可能"

### 1.3 需要的解决方案特征

1. **最小可验证单元**：把诊断知识拆到不能再拆的颗粒
2. **可组合**：基础单元组合出复杂诊断（不是穷举枚举）
3. **可证伪**：每条规则强制写"在什么情况下我错"
4. **可演化**：知识随版本/硬件变化，旧规则不直接覆盖
5. **可审计**：每个结论能复算到客观证据
6. **可教学**：人和 AI 共用同一套规则，能解释为什么

这就是 **原子知识 + 因果推论 + 知识工程**框架要解决的问题。

## 二、整体设计哲学

### 2.1 反 5 大偏见原则

#### 1. 事实 / 推断 / 结论分级

| 层级 | 可写法 | 禁止写法 |
|---|---|---|
| **事实** | "命令显示 X" / "日志记录 Y" | "X 导致问题" |
| **推断** | "Z 可能解释 X" | "一定是 Z 导致 X" |
| **结论** | "在本 case 中最强证据链支持 X → Y" | "以后看到 X 就是 Y" |

#### 2. 正常机制 First（科学方法论的零假设）

> 任何异常告警 / 保护 / 状态码，都不能直接等同于"故障"。先回答：
> 1. 这个信号在设计上什么情况下会**正常**出现？
> 2. 需要哪些上下文证据，才能说它**不该**出现？
> 3. 哪些反证会把"异常候选"降回"正常保护"？

例：`Module lifted` 信号只证明系统进入抬起保护链路。
- 如果确实有人抬起 → 这是正常保护
- 只有在工作移动中、姿态/现场不支持真实抬起时，才是异常候选

#### 3. 共现 ≠ 因果

- "GPS 弱 + Docker 失败"共现不能写成"Docker 解释 GPS"
- 必须有**中间链路证据**（如整机离线、容器未启动、App 状态映射）
- 这是 Pearl 因果推理的核心：**关联（correlation）≠ 因果（causation）**

#### 4. 修复成功 ≠ 因果链正确

- "重启了，机器好了" 不能证明"故障就是 X"
- 修复有效**只证明该动作有效**，不证明诊断链完整
- 必须独立验证因果链

#### 5. 症状 ≠ 根因

- 用户能看到的（"机器不动"）是**症状层**
- 根因可能在电源 / 软件 / 网络任意层
- 必须分层归因，**禁止跨层跳跃**

### 2.2 设计灵感来源

| 学科 / 方法 | 对框架的影响 |
|---|---|
| **Judea Pearl 因果推理**（Causal Inference）| 因果图、节点、边、Do-calculus、反事实必填 |
| **Karl Popper 可证伪性** | "does_not_prove" / "falsified_by" 必填字段、可被反驳即知识 |
| **CS 形式化方法**（Atomic predicates）| 原子知识设计、单一 claim 原则 |
| **临床鉴别诊断**（Differential Diagnosis）| 症状层 → 鉴别清单 → 排除候选、Fault-mode 的 atom_roles 5 类 |
| **Google SRE Postmortem culture** | Blameless review、修复 ≠ 根因、Eval 回归测试 |
| **贝叶斯更新**（Bayesian Update）| 候选假设动态升降权机制 |
| **Test-Driven Development** | 每个 active playbook 必须绑 evals/ 回归测试 |
| **半监督机器学习 candidate 池** | 知识生命周期 candidates → reviewed → accepted |

## 三、5 层知识结构

### 3.1 全貌图

```
┌─────────────────────────────────────────────────────────────────┐
│  Diagnostic-Pack（运行时边界 / 安全围栏）                        │
│   ↑ 编排 atom / edge / fault-mode / playbook / eval              │
│   └────────────────────────────────────────────────────────────┐ │
│   ┌────────────────────────────────────────────────────────────┘ │
│   │ Playbook（诊断流程 / 结构化 SOP）                            │
│   │   ↑ 每步 validates_atoms / validates_edges                  │
│   │   └────────────────────────────────────────────────────────┐│
│   │   ┌────────────────────────────────────────────────────────┘│
│   │   │ Fault-Mode（故障模式 / 病种字典）                       ││
│   │   │   ↑ required_atoms + atom_roles 5 类                    ││
│   │   │   └────────────────────────────────────────────────────┘│
│   │   │   ┌────────────────────────────────────────────────────┘│
│   │   │   │ Causal-Chain（因果链 / 跨层归因图）                 │
│   │   │   │   ↑ 节点（symptom/fact/mechanism/...）+ 带证据强度的边│
│   │   │   │   └──────────────────────────────────────────────────│
│   │   │   │   ┌──────────────────────────────────────────────────│
│   │   │   │   │ Edge（原子关系 / 二元规则）                       │
│   │   │   │   │   ↑ from_atoms → to_atoms（5 类 relation）       │
│   │   │   │   │   └──────────────────────────────────────────┐   │
│   │   │   │   │   ┌──────────────────────────────────────────┘   │
│   │   │   │   │   │ Atom（原子知识 / 最小判断规则）              │
│   │   │   │   │   │   - signal / proves / does_not_prove / scope │
│   │   │   │   │   └──────────────────────────────────────────────│
└─────────────────────────────────────────────────────────────────┘

辅助资产：
- Knowledge-Candidate（候选知识，未审核）
- Knowledge-Card（领域知识卡片）
- Learning-Card（人读心智模型）
- Eval-Case（回归测试）
```

### 3.2 每层的角色与产生顺序

| 层 | 角色 | 产生顺序 |
|---|---|---|
| **Atom** | "看到 X 最多能推出 Y，不能推出 Z" | 第 1 步沉淀（最底层）|
| **Edge** | "Atom A 与 Atom B 的关系（5 类 relation）" | Atom 稳定后才能建 |
| **Causal-Chain** | "症状 → 事实 → 机制 → 假设 → 残留问题"的图 | 多 case 验证后沉淀 |
| **Fault-Mode** | 病种字典条目（症状 + 必要证据 + 区分 atom） | Edge + Atom 库支撑后才能定义 |
| **Playbook** | 结构化 SOP（取证步骤）| Fault-Mode 定义后写 |
| **Diagnostic-Pack** | 运行时边界（编排 + 禁止规则）| 全部就绪后编译 |

### 3.3 上层只能消费下层（核心约束）

```
不允许：
  - Fault-Mode 凭空声明根因（必须 require atom + edge）
  - Playbook 凭空声明结论（必须 validate atom / edge）
  - Pack 引入新知识（必须编排已审核资产）
  - Causal-Chain 用"共现"做因果边（必须用 direct / chain / exclusion）
```

→ 这是反"LLM 自由发挥"的硬约束。

## 四、原子知识库的精髓

### 4.1 一个 atom 长什么样

```yaml
id: ATOM-ROS2-001
title: "ROS2 topic 的 publisher count 只证明端点发现"
kind: evidence_rule           # first_principle / implementation_rule / evidence_rule / diagnostic_recipe
layer: ros2
status: accepted              # hypothesis / accepted / deprecated
statement: |
  `ros2 topic info <topic> --verbose` 返回 Publisher count > 0
  证明 DDS discovery 当前看到至少一个 publisher endpoint。
deterministic_scope: "ROS2/DDS topic introspection at sampling time"
scope_level: version_local    # case_local / site_local / model_local / version_local / global
evidence_rule:
  required_observation:
    - "`ros2 topic info <topic> --verbose` returns Publisher count > 0"
  interpretation: "Topic has a discovered publisher endpoint at sampling time"
  confidence_when_seen: high

does_not_prove:               # ⭐ 反过拟合核心字段
  - "The publisher is emitting fresh data"
  - "The data content is valid"
  - "Downstream subscribers are processing the data correctly"

counter_evidence:             # ⭐ 反驳条件
  - "Publisher count is 0"
  - "DDS discovery is known unhealthy"

source_cases: ["25210102E5C1H969-2026-04-26"]
last_verified: 2026-04-26
review_after: 2026-07-26
```

### 4.2 `does_not_prove` 字段：反过拟合的核心

这个字段是 framework 最关键的设计创新之一：

> **每个 atom 必须强制写"在什么情况下，这条规则不成立 / 不能推出什么"。**

为什么必填：
1. 防 LLM 看到信号就推全局结论（"看到 hg0 Not connected 就说 HaLow 模组坏"）
2. 防跨域误用（24 平台的 atom 直接套到 25 平台）
3. 防症状当根因
4. 防修复偏倚

类比：把 [科学的可证伪性](https://zh.wikipedia.org/wiki/可證偽性) 嵌进数据结构。

### 4.3 Scope 5 级（防过度泛化）

| Level | 适用 |
|---|---|
| `case_local` | 只在当前 case 同一证据窗口内成立 |
| `site_local` | 只在某客户、场地或环境上下文成立 |
| `model_local` | 只在某型号或硬件族成立 |
| `version_local` | 只在源码 / 固件 / 协议版本匹配时成立 |
| `global` | 来自接口定义、物理约束或跨版本验证 — **默认不允许从单 case 升 global** |

→ 单个 case 沉淀的 atom 默认是 `case_local` 或 `hypothesis`，需要跨多 case 验证才能升到 `version_local` / `global`。

### 4.4 原子粒度规范

**合格 atom**：
- ✅ 只表达**一个 claim**
- ✅ 能被客观 evidence 支持或反驳
- ✅ 有明确 `deterministic_scope` 和 `scope_level`
- ✅ 有 `does_not_prove`
- ✅ 有 `counter_evidence` 或 `falsified_by`
- ✅ 不混合观察、原因、修复动作和概率判断

**不合格 atom（典型反例）**：
```yaml
# ❌ 错误的 atom
claim: "回桩灯不灭通常是充电识别失败，应该检查触点。"
# 问题：混合症状（灯不灭）+ 推测原因（充电识别失败）+ 修复建议（检查触点）
```

**正确拆法**：
```yaml
atom_1: "当前窗口内机器人已完成回桩"           # 事实
atom_2: "当前窗口内系统未观测到充电识别成功"   # 事实
atom_3: "充电识别失败与触点接触异常相容"       # edge 候选
atom_4: "触点检查是低风险人工检查"             # 风险评估
```

### 4.5 与传统知识表示的区别

| 维度 | 传统 SOP / FAQ | Atom 知识库 |
|---|---|---|
| 颗粒度 | 段落 / 整页 | 单一 claim |
| 反驳机制 | 无 | 必填 `does_not_prove` + `counter_evidence` |
| 适用范围 | 隐式 | 显式 `scope_level` |
| 可组合 | 不可（each is monolithic）| 可（atom 组合成 edge / causal-chain）|
| 版本管理 | 改文档覆盖 | 改 status → hypothesis / deprecated，保留历史 |
| 审计 | 无 | 必填 `source_cases` / `last_verified` / `review_after` |

## 五、因果推论的关键机制

### 5.1 三层因果模型

任何新问题必须先拆三层：

| 层 | 问题 | 输出 |
|---|---|---|
| **机制层** | 系统设计上什么情况会产生这个信号？ | 正常机制 / 保护机制 / 状态机路径 |
| **上下文层** | 当前场景是否满足正常触发条件？ | 正常触发 / 异常候选 / 证据不足 |
| **根因层** | 上下文矛盾时哪些原因能解释？ | 候选根因树 + 下一步验证 |

### 5.2 证据强度 4 类

| 强度 | 定义 | 用途 |
|---|---|---|
| **直接证据** | 命令/日志直接显示事件 | 可支持事实 |
| **链路证据** | 时间顺序 + 上下游机制连接两个事实 | 可支持候选因果 |
| **排除证据** | 证明某条路径不成立 | 用于降低候选 |
| **共现证据** | 同时间窗出现但**无机制连接** | **只能提示排查，不能归因** |

→ "**共现证据不能升级成因果**"是硬规则。Edge / Causal-Chain 的因果性关系（`causes` / `contributes_to` / `amplifies`）**禁止使用 `co_occurrence` 证据强度**。

### 5.3 因果链节点 5 类

| Kind | 含义 |
|---|---|
| `user_symptom` | 用户看到的现象 |
| `fact` | 日志/命令直接证明的事实 |
| `mechanism` | 系统机制或版本相关行为 |
| `hypothesis` | 待验证假设 |
| `residual_question` | **当前证据不能解释的问题（强制保留）** |

### 5.4 因果链边 8 类

| Relation | 含义 | 可用证据强度 |
|---|---|---|
| `causes` | A 导致 B | direct / chain / exclusion（**禁** co_occurrence）|
| `contributes_to` | A 部分贡献于 B | 同上 |
| `amplifies` | A 放大 B | 同上 |
| `explains` | A 解释 B | direct / chain |
| `weakens` | A 减弱 B | direct / chain |
| `excludes` | A 排除 B | exclusion |
| `co_occurs_with` | A 与 B 同时间窗出现（**无机制**）| co_occurrence |

### 5.5 反事实必填（counterfactual）

每条候选因果边必须回答：
- 如果 A 发生，B 为什么会发生？
- 如果 A 不发生，B 是否还可能发生？
- 什么证据会**推翻**这条边？
- 这条边在什么版本/硬件/环境中适用？

> 没有反事实的边只能是"线索"，不是"因果"。

### 5.6 概率升降权动态更新

每获得一组新证据后，必须更新假设升降权：

| 字段 | 要求 |
|---|---|
| 当前最强方向 | 只写候选，不写最终根因（除非已满足结论门槛）|
| 升权证据 | 哪些新证据让它更可能 |
| 降权证据 | 哪些证据让其他解释更不可能 |
| 仍未排除 | 哪些正常解释或替代假设还活着 |
| 下一步最小证据 | 哪个只读证据最能升降权 |
| 风险边界 | 是否仍 0 风险；非只读必须请示 |

### 5.7 替代假设强制保留（3 类）

诊断输出至少保留：
1. **正常机制**：真实保护、正常策略、人工操作
2. **上报机制**：App stale、云端滞留、错误清除延迟
3. **观测机制**：bag 缺 topic、日志重复、采样窗口错位、字段语义版本不匹配

→ 只有被证据排除或降权后，才能从下一步计划里移除。

## 六、防过拟合的工程化设计

Framework 把"反过拟合"做成了**多层硬约束**：

### 6.1 字段层硬约束

| 约束 | 落实在 |
|---|---|
| 原子知识必填 `does_not_prove` | Atom schema |
| 原子知识必填 `counter_evidence` 或 `falsified_by` | Atom schema |
| 因果性关系禁用共现证据 | Edge + Causal-Chain schema |
| 单一 claim 原则 | Atom schema |
| 因果链必填 `does_not_claim` | Causal-Chain schema |
| 因果链未解释症状必须保留 | Causal-Chain schema |

### 6.2 生命周期硬约束

| 约束 | 含义 |
|---|---|
| 单 case 默认 hypothesis | 单次成功只能假设，不能升 accepted |
| accepted 需 ≥ 2 独立 case 或 1 受控复现 + 反例 review | 升级门槛 |
| 修复成功 ≠ 因果正确 | 不能因为修好了就升 accepted |
| 候选必经 reviewing 才能 accepted | 防 LLM 自动幻觉沉淀 |
| 旧知识降级不直接删除 | 保留历史，写明 superseded_by |

### 6.3 流程层硬约束

| 约束 | 触发 |
|---|---|
| 复杂诊断必经 16 步固定顺序 | governance SKILL |
| 必须先定义正常机制再判异常 | reasoning principles |
| 远程优先 6 优先级 | 不轻易要求客户配合 |
| 时间窗纪律（不同时间窗不互相覆盖）| reasoning principles |

### 6.4 运行时硬约束（Diagnostic-Pack）

| 约束 | 含义 |
|---|---|
| `forbidden_claims` | Agent 在该场景下禁止得出的结论 |
| `forbidden_actions` | Agent 在该场景下禁止执行的动作 |
| `atom_roles.insufficient_alone` | 这些 atom 单独命中不能 confirm 故障 |
| `atom_roles.excluding` | 这些 atom 出现则降级 / 排除主假设 |
| Eval 回归测试 5 类全覆盖 | P0/P1 failed 阻塞 skill 发布 |

## 七、反模式（Anti-Patterns）

### 7.1 反模式的本质

> 反模式 = 真实诊断中走过的弯路，沉淀为"下次遇到类似输入时必须避免的推理错误"。
>
> 反模式**不是根因知识**（不声称"某现象必然由某原因导致"），而是**约束诊断流程**的元规则。

### 7.2 反模式必填字段

每条反模式包含：
- **诱因**：什么场景容易引发这种错误
- **错误跳跃**：人/AI 容易做出的错判
- **正确动作**：应该补什么证据 / 拆什么层
- **适用边界**：什么情况下规则不适用
- **来源案例**：从哪个 case 沉淀的

### 7.3 7 条通用反模式（来自工业实践）

| ID | 名称 | 核心错误 |
|---|---|---|
| **AP-001** | 把用户/App 文案直接等同于底层根因 | 用户报"GPS 弱" → 直接诊断 RTK |
| **AP-002** | 只看容器运行，忽略编排/生命周期健康 | `docker ps` Up → 认为正常，漏看上层 |
| **AP-003** | 把空闲采样过拟合到工作复现路径 | 空闲正常 → 认定工作失败不是该模块 |
| **AP-004** | 把源码机制当成运行版本铁证 | 本地源码能解释 → 认为运行二进制完全一致 |
| **AP-005** | 忽略时区和日志命名规则 | UTC 当本地时间 → 因果顺序判断错 |
| **AP-006** | 低估远程链路和工具本身的失败模式 | 命令失败直接归到设备，不验证工具链 |
| **AP-007** | 把 host/container 能力拼接成不存在的执行环境 | 假设 host 工具在容器内可用 |

### 7.4 反模式 vs 根因知识

| | 反模式 | 根因知识 |
|---|---|---|
| 声称 | "不要这样推" | "X 导致 Y" |
| 升级门槛 | 单 case 即可沉淀 | 需 ≥ 2 case 验证 |
| 沉淀位置 | `knowledge/reasoning/diagnostic-anti-patterns.md` | `knowledge/atoms` / `knowledge/causal-chains` |
| 与现场冲突时 | 以现场证据为准 + 反例 review | 以现场证据为准 |

## 八、知识生命周期与治理

### 8.1 3 状态枚举

| 状态 | 含义 | 使用方式 |
|---|---|---|
| `hypothesis` | 有案例支持，但样本不足或边界不清 | 只能作为排查方向 |
| `accepted` | 有明确证据、版本边界、复用成功记录 | 可作默认判断依据 |
| `deprecated` | 被反例推翻 / 版本过期 / 已有替代 | 不用于诊断，只保留迁移说明 |

### 8.2 候选知识晋升路径

```
诊断发现新经验
    ↓
candidates/pending/  ← 默认进这里，不被信任
    ↓ review / dedup / conflict check
    ↓
candidates/reviewed/
    ↓ accepted artifact diff
    ↓ eval + lint
    ↓
knowledge/atoms / edges / fault-modes / playbooks  ← accepted
    ↓ 反例出现 / 版本过期
    ↓
deprecated/  ← 不删除，保留历史
```

### 8.3 升级到 accepted 的 9 个最低条件

1. 至少一个结构化 case verdict 支持
2. 证据要求客观可检查（不依赖专家口头判断）
3. 适用版本边界明确
4. 至少一个反例或 counter-evidence 条件被写清楚
5. 跨层归因必须有对应因果链，关键边不是共现
6. 至少一道题库题验证关键因果边（不是只验证最终结论）
7. 修复成功不能单独作为升级依据
8. 自动生成的 candidate 必须经过 reviewing
9. 影响诊断行为的 playbook/tool 更新必须新增或更新 evals 并通过 lint

### 8.4 Review 触发

- `review_after` 到期
- 出现强反例
- 相关硬件 / OS / 容器 / 业务版本变更
- 题库答题持续暴露误判
- 新 case 使用该知识但最终结论被推翻
- `candidates/pending/` 中出现相同或冲突候选
- active playbook 关联的 P0/P1 eval 失败

### 8.5 降级流程

发现反例不直接覆盖旧知识：

1. 把旧知识改成 `hypothesis` 或 `deprecated`
2. 在 `review_notes` 写明反例和原因
3. 如果有替代规则，设置 `superseded_by`
4. 更新相关题库，防止继续训练错误模式

→ 这是**"知识图谱永远保留历史"**的设计，与"软件版本回滚"思路一致。

## 九、运行时：Pack 与 Agent 边界

### 9.1 现象 → Pack → Playbook 链

```
用户报告：症状 S
    ↓ select-diagnostic-pack "<S>"
    ↓
匹配候选 pack 列表（按历史命中率排序）
    ↓
进入 pack[0]：
  · 主假设 fault-mode F
  · 取证 playbook P
  · 区分 atom（atom_roles 5 类）
  · 边界（forbidden_claims / forbidden_actions）
    ↓ 跑 playbook
    ↓ 每步 validates_atoms
    ↓
atom_roles 投票：
  ├── required 都命中 → F 被支持 → 输出 verdict
  ├── excluding 出现 → F 被排除 → 切到 pack[1]
  └── 证据不足 → 输出"证据不足"
```

### 9.2 Pack 的本质：现象的"鉴别诊断容器"

| 角色 | 类比医学 |
|---|---|
| 症状 | "咳嗽" |
| Pack | "咳嗽对应的鉴别诊断方案" |
| Pack 主 fault-mode | "肺炎"（最常见 / 主假设）|
| 鉴别 atom | 拍 X 光 / 听诊 / 验血（区分肺炎 vs 哮喘 vs 支气管炎）|
| Playbook | 完整的检查 SOP |
| forbidden_claims | "不能在没拍 X 光时下肺炎诊断" |
| forbidden_actions | "不能在确诊前给抗生素" |

### 9.3 Pack 设计的关键约束

1. **Pack 不能引入新知识** — 只能编排已审核的 atom / edge / fault-mode / playbook
2. **Pack 必须列 forbidden_claims** — 明确 agent 在该场景下禁止得出的结论
3. **Pack 必须列 forbidden_actions** — 明确禁止的操作（如客户机器上播放 bag）
4. **Pack 是编译产物** — 知识在底层资产里维护，pack 只是组合视图

### 9.4 Eval 回归测试 5 类

每个 Fault-mode 最终应有 5 类 eval 覆盖：

| 类型 | 测什么 |
|---|---|
| **positive** | 正例（命中该 fault-mode）|
| **counterexample** | 反例（看似但不是该 fault-mode）|
| **evidence_insufficient** | 证据不足（能正确输出"不够"）|
| **risk_boundary** | 风险边界（能拦截危险动作）|
| **historical_misdirection** | 历史误判（之前走过弯路的复现）|

→ P0/P1 active eval failed → 阻塞 skill 发布。

## 十、与传统 SOP 的本质区别

| 维度 | SOP 决策树 | 原子知识 + 因果推论 Framework |
|---|---|---|
| **表达单元** | 段落 / 流程节点 | 原子（单一 claim）|
| **组合方式** | 树形枚举 | 图组合（atom + edge + causal-chain）|
| **覆盖新故障** | 写不进去就盲区 | 走 fallback + 沉淀 candidate |
| **反过拟合机制** | 无 | `does_not_prove` + `forbidden_claims` + `excluding atoms` |
| **可演化** | 改文档覆盖 | hypothesis → accepted → deprecated 三态生命周期 |
| **跨域复用** | 不能（每域独立树）| 跨域复用 atom |
| **审计能力** | 弱（路径不显式列）| 强（每步可复算到客观证据）|
| **启动成本** | **低**（写一条 SOP = 5 分钟）| **高**（写完整 atom = 30 分钟 + 审核）|
| **启动期 ROI** | 高（立即可用）| 低（需要沉淀库）|
| **长期可扩展性** | 低（规模膨胀）| 高（图组合无限）|

→ 两者**不是替代关系，而是分阶段最优解**：
- **启动期** SOP 决策树 ROI 高
- **规模化阶段** 原子知识库可扩展性更好
- **理想方案**：把已沉淀的 SOP 作为 framework 的 fault-mode + playbook 初始填充

## 十一、应用领域

这个框架不局限于机器人诊断，可应用于**任何"高风险 + 多假设 + 需可审计"**领域：

| 领域 | 应用形式 |
|---|---|
| **机器人 / 物联网售后诊断** | 本框架原始应用领域 |
| **IT 运维 / SRE** | 替代传统 runbook 决策树 |
| **医学鉴别诊断** | atom = 症状/体征解读规则，pack = 病种鉴别 |
| **法律推理** | atom = 法条客观判定，causal-chain = 事实链 |
| **金融风控** | atom = 单一信号判定，pack = 欺诈类型鉴别 |
| **安全事件响应** | atom = 单一指标 IOC，pack = 攻击模式鉴别 |
| **科学实验复现** | atom = 单一观测，causal-chain = 假设验证 |

**共同特征**：
- 错误诊断成本高（不能瞎猜）
- 现象 → 根因有多种可能（需鉴别）
- 需要审计 / 复算 / 学习能力
- 知识需要随版本演化

## 十二、设计的核心创新点

### 12.1 `does_not_prove` 作为一等公民

把 [Popper 可证伪性](https://zh.wikipedia.org/wiki/可證偽性) 嵌进数据结构：**没有 does_not_prove 字段的知识不允许存在**。

### 12.2 原子粒度 + 图组合

不是大段教程，不是决策树，而是**最小可验证单元 + 显式关系图**。类似化学：基础元素有限，但组合无限。

### 12.3 候选池 + 晋升流程

借鉴半监督学习的 candidate 机制：自动生成的经验**默认不可信**，必须经过 review checklist 才能升 accepted。防 LLM 幻觉污染知识库。

### 12.4 反模式作为元规则

反模式**不声称根因**，只约束"下次遇到这种输入不要这样推"。是**反偏见的工程化**，与 governance 流程并列。

### 12.5 Pack 作为运行时围栏

`forbidden_claims` / `forbidden_actions` 是 agent 在特定场景下的**硬边界**，防止 LLM 自由发挥。

### 12.6 Eval 5 类覆盖

回归测试不仅测"得出正确结论"，还测"不得出错误结论 / 不漏风险 / 不做危险动作"。

## 十六、总结：一页核心思想

> **传统知识管理**：把经验写成文档段落（SOP / FAQ / Wiki）。问题：颗粒度粗、无反驳机制、不可组合、不可演化。
>
> **决策树系统**：把诊断逻辑写成 if-else 树。问题：覆盖不全、单 case 过拟合、跨域不复用、缺乏反例。
>
> **原子知识 + 因果推论 Framework**：把知识拆到最小可验证单元（atom），强制每个单元写"不能推出什么"（does_not_prove），用图组合而非枚举（atom + edge + causal-chain），用候选池 + 审核晋升防 LLM 幻觉，用 Pack + forbidden_claims/actions 约束 agent 运行时边界，用 Eval 5 类覆盖防退化。
>
> **核心创新**：把 [Popper 可证伪性](https://zh.wikipedia.org/wiki/可證偽性) + [Pearl 因果推理](https://en.wikipedia.org/wiki/Causal_inference) + [临床鉴别诊断](https://zh.wikipedia.org/wiki/鑒別診斷) + [SRE Postmortem culture](https://sre.google/sre-book/postmortem-culture/) 嵌进数据 schema 和工作流，让"反偏见"从口号变成强制约束。
>
> **适用领域**：任何"高风险 + 多假设 + 需可审计"的诊断 / 推理 / 决策场景。
>
> **代价**：启动期成本极高（写一条完整 atom 需 30 分钟 + 审核，远超写一条 SOP）。Yarbo 实施现状显示，**framework 设计完美 vs 实施层只填了 5%**，需要长期积累才能见效。短期最优策略是用 SOP 决策树作为 framework 的初始填充（fault-mode + playbook），逐步沉淀 atom 库。
