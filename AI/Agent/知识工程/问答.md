# 诊断 Framework 问答整理

> 围绕  Framework（原子知识 + 因果推论 + 知识工程）的一次答疑梳理。按提问顺序整理，保留校准过程。

---

## Q1：Diagnose 框架的设计逻辑是什么？

**A**：**反 LLM / 人类共同诊断弱点的工程化系统**。

把诊断推理拆到**最小可验证单位**（atom），用**因果图组合**而非决策树枚举，强制每条结论标注"不证明什么"、"可被反驳的条件"和"适用范围"。

### 5 层知识结构

```
Diagnostic-Pack（运行时边界）
  ↑ 编排
Playbook（诊断流程 / 结构化 SOP）
  ↑ 每步验证
Fault-Mode（故障模式 / 病种字典）
  ↑ 必要 atom + edge
Causal-Chain（因果链 / 跨层归因图）
  ↑ 节点 + 边
Edge（原子关系 / 二元规则）
  ↑ from_atoms → to_atoms
Atom（原子知识 / 最小判断规则）
```

### 反 5 大偏见原则

1. **事实 / 推断 / 结论分级**
2. **正常机制 First**（先定义正常，再判异常）
3. **共现 ≠ 因果**
4. **修复成功 ≠ 因果链正确**
5. **症状 ≠ 根因**

灵感来源：Pearl 因果推理 + Popper 可证伪性 + 临床鉴别诊断 + SRE Postmortem + Bayesian Update。

## Q2：Framework 这么"牛"，为什么没达到 SOP 的诊断效果？

**A**：**设计完美 vs 实施只填了 5%**。

### 6 个落差点

1. **5 层结构 BI agent 只跑了第 0 层**（数据采集 + 阈值标注）
2. **Atom 库严重不足**（B01733 那种"扫雪段右刀盘 v0.0 = 默认值"的 atom 根本没沉淀）
3. **Fault-mode 库几乎空白**（仅 2 个 accepted）
4. **SOP 知识没沉淀进 framework**（散落在 dev 头脑 / BI 评论）
5. **Bag 数据没接入**（agent 只静态读，不会主动拉 bag）
6. **触发率不足**（chaige 5-1~5-5 仅 30%）

### 鸡生蛋问题

```
要 framework 跑出诊断效果
    ↓
需要丰富的 atom + fault-mode + playbook 库
    ↓
需要从大量 case 沉淀
    ↓
需要 framework 实际跑起来产生反馈
    ↓
但 framework 现在跑得像"残次品"
    ↓
回到起点
```

### 类比

| 层 | 应该有 | 实际有 |
|---|---|---|
| Framework 设计 | 化学元素周期表 + 反应规则 | ✅ 完整 |
| Atom 库 | 100+ 元素 | 3-5 个 accepted |
| Fault-mode 库 | 100+ 化合物 | 2 个 |
| Playbook 库 | 50+ 生产流程 | 2 个 |

→ **理论会用周期表，实际工具只够做 pH 试纸级测试**。

## Q3：Framework 知识能自动补充吗？给我 SOP 能转 framework 吗？

**A**：**第三个判断完全正确** — Atom 是基础颗粒，SOP 是组合应用，给 AI SOP 它能反推 fault-mode + playbook 骨架，但 atom 的核心字段（does_not_prove / scope）AI 写不出来。

### AI 能自动转换的（约 70%）

| 转换 | AI 能力 |
|---|---|
| SOP → fault-mode 骨架 | ✅（症状 / 涉及组件 / 处置 都在 SOP 里）|
| SOP → playbook 步骤 | ✅（SOP 步骤直接转）|
| SOP → atom 的 `signal` 字段 | ✅ |
| SOP → atom 的 `proves` 字段 | ✅ |
| 跨 SOP → 候选 edge | ⚠️ 部分（统计同时出现的 atom）|

### AI 不能自动做（必须人工，约 30%）

| 转换 | 为什么 AI 不能 |
|---|---|
| atom 的 **`does_not_prove`** | SOP 只说"X → Y"，不说"X 不能推出 Z" |
| atom 的 **`scope`** | SOP 不写适用边界；需多 case 反例才能划范围 |
| edge 的 **反证条件** | 需积累多个失败案例 |
| **盲区 atom**（"扫雪段右刀盘 v0.0 = 默认值"）| SOP 只写故障，不写"看似异常但实际默认值" |

### 关键洞察

> SOP 是 atom 的"组合应用"，不是 atom 本身。
>
> 给 AI SOP，AI 能反推 fault-mode + playbook 框架，但 **`does_not_prove`、`scope`、反例条件 必须人工补**。
>
> **`does_not_prove` 是 AI 的盲区** — 需要领域工程经验 + 多 case 反例验证 + 跨域知识。

## Q6：原子关系（Edge）和因果链（Causal-Chain）的区别是什么？

**A**：

| 维度 | Edge | Causal-Chain |
|---|---|---|
| **规模** | 1 关系 = 2 组 atom 之间 | N 节点 + M 边的图 |
| **端点性质** | 两端都是 atom（客观信号规则）| 5 类节点（**含 user_symptom + residual_question**）|
| **relation 数** | 5 类（prerequisite_for / supports / constrains / excludes / co_occurs_with）| 8 类（多 causes / contributes_to / amplifies / explains / weakens）|
| **必含 scope** | 否（atom 自带）| ✅ 必须，含 `does_not_claim` |
| **必含未解释项** | 否 | ✅ 必须有 `residual_question` |
| **升级到 accepted** | 单 case 可（有反例条件即可）| **≥ 2 独立 case 或 1 受控复现 + 反例 review** |

### 类比

| 类比 | Edge | Causal-Chain |
|---|---|---|
| 交通 | 道路（A 城到 B 城路线规则）| 一次具体行程（从家到医院走哪些路 + 哪些没走过）|
| 化学 | 化学键（H-O 共价键）| 完整反应方程式（H₂+O₂→H₂O，含起始物 + 产物 + 反应条件）|
| 语言 | 主谓关系 | 一整段论证（论点 + 证据 + 反驳 + 未解决问题）|

### 引用关系

```
EDGE-M25-001 (急停不一致 supports M25 接触不良)
    ↑
    ├── 被 CC-M25-BLADES-NOT-READY-001 引用
    ├── 被 CC-M25-EMERGENCY-STOP-002 引用
    └── 被 CC-DISP-RTK-LOSS-003 引用
```

→ Edge 是**可复用的关系组件**，Chain 是**面向具体故障的完整推理**。

## Q7：原子关系是静态知识，因果链是出现问题进行诊断才串起来的？

**A**：**理解基本对，但要校准两点**。

### 校准 1：Chain 也是长期沉淀知识，不是 case 一次性产物

| | 一次性 case 产物 | 长期沉淀知识 |
|---|---|---|
| 位置 | `cases/<SN>-<DATE>/REPORT.md` | `knowledge/causal-chains/*.yaml` |
| 复用 | 单 case 不复用 | accepted 后可复用 |
| 升级门槛 | 不需要审核 | **≥ 2 独立 case + 反例 review** |

→ 单 case 的诊断报告**引用** chain，但 chain 本身是**多 case 共有推理被提炼后固化**。

```
case 1 诊断 → 提出 chain hypothesis
case 2 诊断 → 验证 hypothesis
反例 review → 升 accepted causal-chain
case 3+ → 直接引用 accepted chain
```

### 校准 2：Edge 也不是"完全静态"，但**不涉及症状**

Edge 也能动态校准（出现新反例会降级），但它的两端都是 **atom**（客观信号规则），**不含 user_symptom 节点**。

### 精确版本

- **Edge** = atom 之间稳定的关系规则（**不涉及任何具体症状**）
- **Causal-Chain** = 从用户症状到根因机制的完整推理图（**含症状 + 含未解释问题**）

## Q8：Diagnostic-Pack 和 Playbook 具体干嘛的？

**A**：

- **Playbook** = "**做什么**"（可执行的诊断步骤流程，等于结构化 SOP）
- **Diagnostic-Pack** = "**这个场景下能做什么 / 不能做什么**"（agent 运行时的安全围栏 + 编排清单）

### 类比

| 角色 | Playbook | Diagnostic-Pack |
|---|---|---|
| 厨房 | 菜谱（具体步骤）| 菜单 + 厨房规则（套餐含哪些菜 + 不能开火超过 X 度）|
| 飞行 | 飞行手册（起飞/巡航/降落动作）| 航线计划 + 飞行限制（不能飞过 X 高度）|
| 软件 | 函数实现 | 接口契约（输入输出 + 不能抛什么异常）|

### Pack 的关键字段（运行时边界）

```yaml
pack:
  fault_mode: <主假设>                  # 用哪个 fault-mode 当主假设
  playbooks: [<playbook>]               # 用哪个 playbook 取证
  required_atoms / atom_roles           # 编排哪些 atom
  edges                                 # 编排哪些 edge
  evals                                 # 绑定回归测试
  forbidden_claims: [...]               # ⚠️ Agent 禁止得出的结论
  forbidden_actions: [...]              # ⚠️ Agent 禁止执行的动作
  first_steps: [...]                    # 入口动作
```

### 为什么需要两个

- **只有 Playbook**：agent 跑步骤但可能在间隙做出未授权决策（缺安全围栏）
- **只有 Pack**：知道能做什么但不知道按什么顺序（缺执行路径）
- **Pack 编排 Playbook + 限制边界** = 完整运行时

## Q9：必须有对应的 Pack 才能开启流程吗？

**A**：**不必须**，但没 pack 就走 **governance fallback**（约束弱）。

### Governance 16 步原文

> 第 3 步：运行 `tools/select-diagnostic-pack "<用户症状>"`；命中的 active pack 必须作为运行时边界读取，**未命中时按 governance fallback 处理**。

### 命中 vs 未命中

| 维度 | 命中 Pack | 未命中（fallback）|
|---|---|---|
| 运行时边界 | `pack.forbidden_claims` / `pack.forbidden_actions` 硬绑定 | 仅 governance 通用禁止表达 |
| 数据采集 | `pack.required_atoms` 明确知道要拉什么 | 通用全链路基线（可能漏关键数据如 bag）|
| 关联 playbook | 自动加载 | 无 playbook，纯靠 agent 自组织 |
| 验证 | `pack.evals` 覆盖 | 无 eval 兜底 |
| **过拟合风险** | 低 | **高** |

### 当前现状

- 现有 pack: **仅 2 个**
- 实际所需: 30-50 个
- **95%+ 的工单走 fallback**

→ 这就是 B01733 这种 case 被误判的核心原因（没 M25 pack 来约束 agent）。

## Q11：Playbook 是什么？给个例子

**A**：

### 真实例子：充电灯不灭只读诊断 Playbook（5 步）

```yaml
playbook_id: playbook.robot_state.charging_light_readonly
version: v1.0.0
status: active

steps:
  - S1: 搜历史 case（验证 ATOM-DIAG-001）
    risk_level: L0
  - S2: 跑只读快照采集（验证 ATOM-LIGHT-001/002 + EDGE-LIGHT-001）
    risk_level: L0
  - S3: 如当前未复现 → 盘点历史 bag（验证 ATOM-MCAP-001）
    risk_level: L0
  - S4: 获批后上传 S3（验证 ATOM-REMOTE-001）
    risk_level: L2   # 需审批
  - S5: 本地解析 MCAP（验证 ATOM-LIGHT-001/002/003 + EDGE-LIGHT-001）
    risk_level: L0

stop_conditions:
  - 任何写操作 / 重启 / 改配置触发
  - 只有当前窗口证据 → 输出"证据不足"

risk_policy:
  default: L0
  require_approval_for: [L2, L3]

evals: [EVAL-CHARGING-LIGHT-001..005]
```

### Playbook 关键特征

1. **流程化**：1/2/3/.../N 步骤明确
2. **每步带风险等级**（L0/L1/L2/L3）
3. **每步要验证 atom/edge**（不是空跑命令）
4. **明确停止条件**（防过度归因）
5. **绑 evals**（回归测试覆盖）

### Playbook 本质

把"该按什么顺序取什么证据、每步该证明什么 atom、何时停"明确写出来的**可执行诊断剧本**。

## Q12：Playbook 不就是 SOP？Pack 下多 Playbook 是因为换备选？

**A**：**理解 100% 对（playbook = SOP），但"多 playbook 换备选"实际不主流**。

### Playbook = 结构化 SOP

| | FSE 的 SOP | 结构化 Playbook |
|---|---|---|
| 本质 | 步骤化排查清单 | 步骤化排查清单 |
| 形式 | Word / Confluence / 头脑 | YAML |
| 可 agent 执行 | ❌ | ✅ |
| 强制约束 | 无 | risk_level / validates_atoms / stop_conditions / evals |

→ **Playbook = 加了 schema 强制约束的 SOP**。

### "跑不出来怎么办" — 3 种处理方式

#### 方式 A：同 playbook 内部分支（最常见）

```yaml
steps:
  - S1: 跑静态读
  - S2: 如果 S1 不够 → 跑动态测试
  - S3: 如果 S2 仍不够 → 拉历史 bag
  - stop_conditions: 只有当前窗口证据 → 主动停 + "证据不足"
```

#### 方式 B：触发新 pack（不同现象）

```
当前 pack 跑完发现是另一个故障域
    ↓
切换到 pack[1]（不同的 fault-mode 假设）
```

#### 方式 C：同 pack 内多 playbook 串行（schema 允许但实际罕见）

```yaml
pack:
  playbooks:
    - playbook.static-readonly      # 先跑（轻量）
    - playbook.bag-offline           # 再跑（中等成本）
    - playbook.dynamic-test          # 最后跑（高成本需授权）
```

### 当前现状

```yaml
pack.charging-light-not-off.v1:
  playbooks: [playbook.charging-light-readonly]   # 只 1 个
```

→ 当前 2 个 pack 都**只有 1 个 playbook**。Schema 允许多 playbook 是给未来留扩展口。

## Q13：故障模式是 Playbook 运行完成后得到的结果吗？

**A**：**方向反了**。Fault-mode 是 playbook 的**待验证假设**，不是输出。

### 正确关系

```
[现有 fault-mode 库]
  ├── fault.charging_light_stays_on   ← 预定义的故障对象（病种字典）
  ├── fault.rtk_gnss.poor_signal_near_dc
  └── ...

  ↑ 被引用 / 验证
  │

[Pack 编排]
  pack.linked_fault_modes = [fault.X]
  pack.linked_playbooks = [playbook.Y]

  ↑ 调用
  │

[Playbook 跑] ← 输入"待验证 fault-mode"
  每步 validates_atoms → 收集 atom 命中数据

  ↓ 输出

[Case Verdict] ← 真正的产出
  "fault-mode X 被支持 / 排除 / 不确定"
```

### 类比医学诊断

| | 医学 | Framework |
|---|---|---|
| 预定义对象 | 病种字典（肺炎 / 哮喘 / 支气管炎）| Fault-mode 字典 |
| 鉴别诊断 | 鉴别清单 | Pack |
| 检查流程 | 拍 X 光 / 听诊 / 验血 | Playbook |
| 病历 | "本次诊断肺炎，排除哮喘" | Case verdict |

### 例外：新 fault-mode 从 case 沉淀

如果 case 跑 fallback 模式发现新故障家族：

```
case 诊断 → 发现新模式
    ↓
进 candidates/pending/   ← 不是 accepted
    ↓
拆 atom_candidate
    ↓
跨 case 验证（≥ 2 case）+ review
    ↓
晋升为 accepted fault-mode
```

→ 这种情况下 fault-mode **是 case 沉淀的产出**，但通过 **governance 流程**沉淀，不是 playbook 直接输出。

## Q14：一个 Pack 一种现象 → 一种现象可能是不同 Fault-Mode 吗？

**A**：**完全对**。这就是 pack 的核心价值 — **"鉴别诊断容器"**。

### 用"充电灯不灭"举例 — 理论上至少 5 种根因

| # | 候选 fault-mode | 故障层 |
|---|---|---|
| 1 | `fault.charging_light_stays_on` | 状态机卡死（软件层）|
| 2 | `fault.led_module_hardware_failure` | LED 模块物理故障（硬件层）|
| 3 | `fault.charging_detection_false_negative` | 充电识别误判 |
| 4 | `fault.battery_current_sensor_drift` | 电流传感器漂移 |
| 5 | `fault.light_control_command_lost` | 灯控命令丢失（通讯层）|

→ 同症状"灯不灭"，**5 种完全不同的故障域和处置**。

### 理想 Pack 通过 atom_roles 鉴别

```yaml
pack.charging-light-not-off.v2 (理想):
```

## Q15：Pack 选定但没分析，怎么知道 Fault-Mode？为什么 Fault 是 Playbook 的输入？

**A**：**我之前措辞错了**。需要校准。

### 校准

| 我之前说 | 准确表述 |
|---|---|
| "fault-mode 是 playbook 的输入" | ❌ 措辞错 |
| 实际 | ✅ **fault-mode 是 pack 的"主假设"，playbook 是为验证这个假设而设计的取证流程** |

### 修正后的关系

```
症状关键词匹配
    ↓
[Pack] = "假设这是 fault-mode X" + "取证用 playbook Y"
                            ↑                       ↑
                       主假设（待验证）        验证流程（预绑死）
    ↓ pack 内 playbook 已定，不需要"选"
    ↓
[Playbook] 跑步骤 → 收集 atom 命中证据
    ↓
[atom_roles 投票]
    ↓
├── required atoms 都命中 → fault-mode X 被支持（假设成立）
├── excluding atom 出现   → fault-mode X 被排除（假设失败）→ 切到下一个候选 pack
└── 证据不足              → 输出"证据不足"
```

### Pack 选定 ≠ Fault-mode 确认

```
症状："充电灯不灭"
    ↓ select-diagnostic-pack
    ↓
匹配到 pack.charging-light-not-off.v1
    ↓
这个 pack 携带：
  · 主假设：fault.charging_light_stays_on
  · 验证 playbook：playbook.charging-light-readonly
  · 边界（forbidden_claims/actions）
    ↓
⚠️ 此时 fault-mode 还未确认，只是"假设方向"
    ↓
跑 playbook → 收证据
    ↓
投票：
  ✅ 假设成立 → 输出 verdict
  ❌ 假设被排除 → 触发下一个 pack
```

### 模式：一现象 → 多 pack 候选（每个 pack 一假设）

```
现象 X
  ↓ select-diagnostic-pack 返回候选（按历史命中率排序）：
    1. pack_A (主假设 fault_A, 历史 70%)
    2. pack_B (主假设 fault_B, 20%)
    3. pack_C (主假设 fault_C, 10%)
  ↓
agent 按顺序跑：
  · pack_A.playbook → 排除？→ pack_B → 排除？→ pack_C
```

### 关系再校准（最终版）

| 资产 | 角色 |
|---|---|
| **Fault-mode** | 故障字典里的**条目**（病种）— 预定义对象 |
| **Pack** | "**一个 fault-mode 假设 + 验证它的工具包**"（病种 + 检查方案）|
| **Playbook** | **取证流程**，可被多 pack 共享 |
| **Atom_roles** | 在 pack 内**投票**判定主 fault-mode 是否成立 |
| **Case verdict** | **投票结果**（playbook 真正的输出）|

## 附录：术语速查

| 术语 | 一句话定义 |
|---|---|
| **Atom** | 最小可验证判断规则（信号 → 最多推出什么 / 不能推出什么）|
| **Edge** | 两组 atom 之间的稳定关系（5 类 relation）|
| **Causal-Chain** | 含症状 + 事实 + 机制 + 假设 + 未解释问题的完整推理图 |
| **Fault-Mode** | 病种字典条目（症状 + 必要证据 + atom 角色分类）|
| **Playbook** | 结构化 SOP（每步带风险等级 + 验证 atom + 停止条件）|
| **Diagnostic-Pack** | 运行时安全围栏（编排 + 禁止规则）|
| **Candidate** | 未审核的诊断经验（默认 pending，不可信）|
| **does_not_prove** | atom 强制必填，防过度解释 |

## 关键认知总结（17 条精华）

1. Framework 是**反 LLM/人类共同弱点的工程化系统**（不是 SOP 改良版）
2. **5 层结构**：atom → edge → causal-chain → fault-mode → playbook → pack
3. **上层只能消费下层**，pack 不能引入新知识
4. **`does_not_prove` 是 atom 必填字段**，防过度解释（schema 硬约束）
5. **共现 ≠ 因果**：因果性关系禁用共现证据强度
6. **Edge 不含症状，Chain 含症状 + 未解释问题**
7. **Fault-mode 是 playbook 的待验证假设，不是输出**
8. **Pack 是"现象的鉴别诊断容器"** — 内部可含多 fault-mode 候选
9. **Playbook 就是结构化 SOP**（加 schema 强制约束）
10. **Pack 选定 ≠ Fault-mode 确认**，playbook 跑完投票才确认
11. **没 pack 走 fallback**（约束弱，容易误判，B01733 印证）
12. **单 case 不能升 accepted**，必须 ≥ 2 case + 反例 review
13. **修复成功不等于因果正确**
14. **AI 半自动转 SOP → framework**：能转 70% 骨架，但 does_not_prove / scope 必须人工
15. **Framework 设计完美 vs 实施 5% 覆盖**：当前 95%+ 工单走 fallback
16. **理论需要 30-50 个 pack**，当前只有 2 个
17. **启动期"鸡生蛋"问题**：写 atom 成本高（30 分钟/条），SOP 决策树短期 ROI 更高
