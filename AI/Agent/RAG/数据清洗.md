数据清洗不是「一个动作」,而是**一组按顺序做的操作**。业界(Databricks、微软等)基本把它拆成了固定的几个组件。一条 RAG 数据管线的关键环节包括:语料组成与摄取、数据预处理、解析、富化、元数据抽取、去重、过滤、分块——其中「清洗」散落在解析之后到分块之前这一段。我按**影响从大到小**给你排个可操作的顺序。

## 核心原则:先治源头,再治细节

最该先做、也最容易被跳过的,是**源头筛选**,而不是埋头洗文本。先明确到底哪些文档/网站/数据库包含 RAG 该访问的知识,要具体(比如"HR 政策文件夹""产品手册 v3.0 以上""工单知识库")。喂错了源,后面洗得再干净也是错的。

## 各项清洗操作(附方法与工具)

|操作|怎么做 / 工具|为什么对 RAG 关键|
|---|---|---|
|**格式统一**|尽量转成一致的机读格式;PDF 要确保是可选中的文字而非图片;Markdown/HTML 往往比复杂 Word/PPT 更易准确解析|解析失败 = 内容进不了库|
|**OCR(扫描件)**|图片型 PDF/扫描件用 OCR 提文字,且务必复查 OCR 质量——这里的错误会直接影响 AI 的理解;工具 Tesseract、unstructured 内置 OCR、云 OCR|乱码进库会污染检索|
|**噪音清除**|去掉或规范化特殊字符,处理段落、页眉、页脚等结构元素;工具:正则、`unstructured` 的 `clean` 系列函数、html2text|页眉页脚反复出现会稀释语义、占满 top-k|
|**去重**|精确去重 + 近似去重(下面给代码)|RAG 最关键的一项,单独讲|
|**版本/时效治理**|删除完全重复的文件;识别并归档或明确标记过期版本,避免检索到冲突或过时的信息|同一问题检索到两个矛盾答案 → 模型乱答|
|**PII 脱敏**|工具:Microsoft Presidio、正则|RAG 工作流优先做噪音清除和 PII 脱敏,以保护隐私并优化检索|
|**规范化**|实体去重、拼写语法纠错、消除冗余、数据归一化,以及针对你文档定制的清洗规则|让相同概念表达一致,嵌入更准|
|**富化 + 元数据**|用额外元数据富化数据并去噪;抽取有用的元数据以实现更快更高效的检索|支撑后续元数据过滤、引用溯源|
|**质量校验**|RAG 虽然基于检索,但依赖源材料正确;对关键文档做高层审查或找领域专家背书|源错了,RAG 只会"自信地答错"|

## 重点讲「去重」——这是 RAG 特有的硬骨头

为什么 RAG 特别在意去重?去重能消除冗余条目,优化向量数据库。更直接的危害是:同一段内容若重复入库,检索时会**占满 top-k 名额**,把本该出现的其他相关内容挤出去。去重分两层:

**精确去重**——内容完全相同。规范化后取哈希即可:

```python
import hashlib

def normalize(text):
    return " ".join(text.lower().split())  # 去多余空白、统一大小写

seen, unique = set(), []
for chunk in chunks:
    h = hashlib.md5(normalize(chunk).encode()).hexdigest()
    if h not in seen:
        seen.add(h)
        unique.append(chunk)
```

**近似/语义去重**——内容相似但不完全一样(比如同一政策的两个改版)。要分析文档,识别并剔除重复或近似重复的文档。两种主流做法:

一是 **MinHash + LSH**(看字面重合度,快、适合海量文本),用 `datasketch` 库;二是**语义去重**(看含义,更准),拿嵌入向量算余弦相似度,超过阈值就判为重复:

```python
# 语义近似去重思路(伪代码骨架)
import numpy as np

embeddings = embed(unique)            # 复用你的嵌入模型
kept = []
for i, emb in enumerate(embeddings):
    # 与已保留项的最大相似度
    if not kept or max(cosine(emb, embeddings[j]) for j in kept) < 0.95:
        kept.append(i)
unique = [unique[i] for i in kept]
```

阈值 0.95 是个起点,自己在数据上调:太低会误删不同内容,太高去不干净。

## 工具落地建议

不用全自己写。组合拳大概是:`unstructured` 负责加载解析 + 基础噪音清理 → 自己加一层去重(`datasketch` 或嵌入相似度)→ `Presidio` 做 PII 脱敏 → 解析时顺手抽元数据。LangChain/LlamaIndex 里也有部分清洗组件可直接接。

## 顺序与「洗到什么程度」

实操顺序:**源头筛选 → 格式统一/OCR → 噪音清除 → 去重 → 时效治理 → 脱敏 → 规范化 → 元数据富化 → 抽样校验**。

至于洗到什么程度,别追求完美——解析时要做好错误处理和日志,快速定位问题;这往往会暴露源数据本身的质量问题。一个务实的判断:先把「源头烂数据、重复、过期、乱码」这四类高危问题清掉(它们就是之前说的 80% 失败根因),跑通后用 RAGAS 评估;如果某类问题召回还是差,再回头针对性地加清洗规则,而不是一开始就把所有规则堆上。

需要的话,我可以把「unstructured 解析 + 去重 + 脱敏 + 抽元数据」串成一个完整的清洗脚本给你,你换上自己的文件路径就能跑。
