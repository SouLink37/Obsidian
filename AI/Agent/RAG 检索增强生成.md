> **Retrieval-Augmented Generation -检索增强生成**。

## 一句话定义

**RAG = Retrieval-Augmented Generation = 检索增强生成**。

翻译人话:**LLM 自己回答 → 不靠谱;先去你的知识库里翻一翻,带着翻到的内容再回答 → 靠谱多了**。

## 解决什么问题

LLM 有两个天生的"残疾":

1. **知识截止**:Claude 4.7 的训练数据停在 2026 年 1 月,你问它 2 月的新闻它不知道
2. **不知道你的私有数据**:你公司的金蝶 ERP、飞书 wiki、客户合同——LLM 一概不知

**靠重新训练模型来解决?**太贵太慢,而且每次更新文档都要重训一遍。

**RAG 的思路**:模型不动,**临时把相关资料塞进 prompt**。这就像考试时不让你背全部知识点,而是让你**开卷考试**——遇到问题先翻书,翻到相关段落抄到答题卡上,再基于这些段落作答。

## 工作流程拆开看

RAG 实际上分两个阶段:

### 阶段 1:离线建库(一次性)

```
原始文档(PDF/Word/网页)
    ↓
切块(Chunking) — 切成 200-500 字的小段
    ↓
向量化(Embedding) — 每段用 embedding 模型转成一串数字(向量)
    ↓
存入向量数据库(FAISS / Pinecone / Milvus / Postgres+pgvector)
```

举个例子,你的 PMC 文档 "海外仓库存口径:实时库存扣除当月预计销售消耗" 这一段,会被转成一个 1536 维的向量,存到向量库里。

### 阶段 2:在线查询(每次问问题)

```
用户问题:"海外仓库存怎么算?"
    ↓
问题向量化 — 同样的 embedding 模型转成向量
    ↓
向量相似度搜索 — 在库里找最像的 Top 5 段
    ↓
把这 5 段塞进 prompt,变成这样:
  
  "根据以下资料回答用户问题:
   【资料 1】海外仓库存口径:实时库存扣除当月预计销售消耗...
   【资料 2】...
   
   用户问题:海外仓库存怎么算?"
    ↓
LLM 基于这些资料生成回答(并能说"根据资料 1...")
```

**关键点**:LLM 没"学会"你的数据,它只是**临时读了一下**。下次提问会重新检索。

## 为什么不直接全塞进 prompt

你可能会想:"我把所有 PMC 文档都拼到 system prompt 里不就行了?"

不行,有三个硬约束:

1. **上下文窗口有限**:Claude 4.7 是 1M tokens,但塞满了之后幻觉率会飙升(那位架构师测出来 40-50% 窗口就是分水岭)
2. **每次调用都贵**:每次问一个简单问题都要重新喂 100 页文档,token 成本直接爆炸
3. **信号被噪音淹没**:相关的那 1 段被无关的 99 段稀释,LLM 反而抓不住重点

**RAG 的本质就是:只把"这次问题真正相关的片段"喂给 LLM,其他都不喂**。

## 一个具体类比帮你彻底搞懂

想象你雇了个新员工(LLM):

|方法|类比|问题|
|---|---|---|
|**不做 RAG**|让新员工凭印象答客户问题|他不知道你公司的事|
|**塞全部资料**|把整箱 100 本手册搬给他每次都翻一遍|慢、累、抓不住重点|
|**微调模型**|送他去 3 个月脱产培训|贵、慢、培训完手册又更新了|
|**RAG**|给他配个秒查的搜索引擎,问问题前先查 3 条相关段落|✅ 快、准、随时更新|

**RAG 是给 LLM 配了个"考试备忘录"**——LLM 自己不变聪明,但回答质量大幅提升。

## RAG 的演化

业界这两年 RAG 快速进化,你应该知道几个层次:

### Naive RAG(朴素 RAG)

最简单:切块 → embed → 检索 Top K → 塞 prompt。**90% 的"AI 知识库""企业问答机器人"都是这个**。

### Advanced RAG(进阶 RAG)

加几道工序:

- **Query Rewriting**:用户问题先让 LLM 改写得更适合检索
- **Reranker**:检索回来的 Top 20 用专门的 reranker 模型再排一次,挑出真正最相关的 Top 5
- **Hybrid Search**:向量检索 + 关键词检索(BM25)结合,弥补纯向量的盲区

### Agentic RAG(智能体式 RAG)

**让 agent 自己决定要不要检索、检索什么、检索几次**:

- 看到问题先想:"这个我不需要查,直接答"
- 觉得需要查:"先查 A 主题,看完结果不够,再查 B 主题"
- 类似人类学者写论文的过程

那位架构师说的"京东物流 AI 原生调度系统通过 Agentic RAG 动态调整仓储布局"就是这种。

### GraphRAG(微软开源的)

不只是切块向量化,**先把文档抽成知识图谱**(实体+关系),检索时既看相似度也看图谱关系。处理"多跳推理"问题更强,比如"和 X 公司有合作的、做半导体的、北美企业有哪些?"

## 哪些场景需要 RAG / 哪些不需要

这是最实用的判断:

**需要 RAG**:

- 企业知识库问答(政策、SOP、产品手册)
- 客服机器人(FAQ、退换货规则)
- 法律/医疗/学术资料检索
- 代码助手查 API 文档(那个超大型项目的"wiki rag"就是)

**不需要 RAG**:

- **结构化数据查询**:你的金蝶 ERP 里"X 物料的库存"——直接 SQL 查最快最准,搞向量化反而引入误差
- **简单事实**:LLM 自己知道的(法国首都是哪)
- **数学/代码生成**:不需要查文档的纯生成任务

## 对你 PMC agent 的具体应用

我之前说过"工具越少,可控性越高"——那位工业大模型开发者从 LangChain 全家桶逐步精简到只保留 FAISS + BGE-small-zh + 量化 8B 模型,最后反而效果最好,因为每一步都完全可解释。这个经验对你也适用。

PMC agent 里 RAG 的**正确用法**:

|数据类型|用 RAG 吗?|用什么|
|---|---|---|
|金蝶库存数据、订单数据|❌|SQL 直查 / MCP|
|飞书审批流程数据|❌|API 直查|
|物料 BOM、规格|❌(结构化)|SQL / 表格查询|
|**6 类需求的业务规则文档**|✅|RAG(规则会更新)|
|**历史会议纪要、决策记录**|✅|RAG|
|**物料备料 SOP、异常处理手册**|✅|RAG|
|**Tee/Leo 历史手算的 case**|✅(作 few-shot 示例)|RAG|

**翻译成行动**:你的 agent 主体是 SQL/API 查询,**RAG 只用来给 agent 补充"业务知识和规则"**。这种"结构化数据 + RAG 知识库"的混合架构是 back-office agent 的标准做法。

## RAG 的常见坑

最后给你预警几个翻车点:

1. **切块大小**:切太小丢上下文,切太大稀释信号。一般 200-500 字 + 50 字 overlap 起步
2. **embedding 模型**:中文场景别用 OpenAI 的英文 embedding,用 BGE-zh 或 m3e 这类中文优化模型
3. **检索质量 ≠ 答案质量**:检索召回了相关段不代表 LLM 一定会用对。要测端到端
4. **Top K 不是越多越好**:K=20 比 K=5 检索"更全",但塞进 prompt 的噪音也更多,效果常常更差
5. **更新一致性**:文档改了,**记得重新 embed 重新入库**,否则 agent 答的还是老版本

## 一句话总结

**RAG 是"开卷考试"——LLM 不变聪明,但每次回答前先去你的资料库翻一翻最相关的几页,基于这些页作答**。它解决的是"LLM 不知道你的私有数据"这一类问题,但**只对非结构化文本数据有用**;结构化数据走 SQL 是更对的路。

你 PMC agent 里 90% 的逻辑不需要 RAG,但**业务规则文档、SOP、历史 case** 这一小部分用 RAG 接进来,会让 agent 的判断质量上一个台阶。
